# Phase1 v4 実装・v4 core sweep メモ

## 1. Phase1 v4 を実装した背景

Phase1 では、まず v1/v3 の Residual World Model を実装・評価した。

- v1/v3 では以下を検証した。
  - residual target
  - local residual
  - write mask
  - dynamic ROI / residual prediction
- world model 単体の評価では、MSE / LPIPS / 一部 ranking 指標は改善した。
- しかし RFT 後の success は `pixel_baseline` からほぼ改善しなかった。
- つまり、dense pixel residual をうまく予測するだけでは、RFT の reward として十分に効かない可能性が高い。
- そこで、RFT に効く action-sensitive future representation が必要になった。
- ResWorld の考え方である temporal residual / sparse dynamic query / future-guided interaction を参考に、Phase1 v4 として `TemporalDynamicQueryResidualWM` を実装した。

## 2. v4 の目的

v4 の主目的は Full MSE を下げることではない。

目的は以下。

- history から動的領域を sparse な dynamic query として抽出する。
- action によって future dynamic query を予測する。
- future dynamic query を current spatial tokens に fuse して未来画像 `pred_future` を出す。
- v4b では `ActionFutureScorer` により `ranking_score` を出す。
- 最終的には RFT で使える action-sensitive ranking signal / future dynamic representation を作る。

したがって v4 core sweep では、MSE/LPIPS だけでなく、`pairwise_acc_score`, `score_gap_mean`, mask/query collapse, negative quality を重視する。

## 3. v4 のモデル構造

入力:

- `pixels`: `[B, K+2+H, H_img, W_img, C]`
- `actions`: `[B, K+1+H, action_dim]`

frame layout:

```text
pixels:
  history  : 0 ... K-1
  context  : K        （現在の実装では model 本体では使わない slot）
  current  : K+1
  future   : K+2 ... K+2+H-1
```

全体の流れ:

```text
pixels
  ↓
PixelEncoder
  ↓
z_seq [B, K+1, N=64, D=256]
  ↓
DynamicQueryExtractor
  ↓
dynamic_queries [B, K+1, Q, D]
dynamic_masks   [B, K+1, Q, N]
  ↓
TemporalResidualPredictor
  ↓
future_dynamic_queries [B, H, Q, D]
  ↓
TokenFuser
  ↓
future_spatial [B, H, N, D]
  ↓
PixelDecoder
  ↓
pred_future [B, H, 3, H_img, W_img]

v4b:
future_dynamic_queries + action_tokens
  ↓
ActionFutureScorer
  ↓
ranking_score [B]
```

## 4. 各モジュールの役割

### PixelEncoder

- history/current 画像を 8x8 spatial tokens に変換する。
- 256x256 image から `N=64` spatial tokens を作る。
- 出力はおおむね `[B, K+1, N=64, D=256]`。

### DynamicQueryExtractor

- current frame から `current_shared_mask` を作り、Q 個の dynamic query を抽出する。
- mask は `[B, Q, N]` で、同じ query index が history/current 間で対応するように shared mask として使う。
- `USE_MOTION_BIAS=1` の場合は `||z_t - z_{t-1}||` を使い、動いている領域に query を寄せる。
- 出力:
  - `dynamic_queries [B, K+1, Q, D]`
  - `dynamic_masks [B, K+1, Q, N]`

### TemporalResidualPredictor

- dynamic query の時間差分から temporal residual query を作る。
- current queries と residual queries を context として使う。
- action tokens と step embedding を合わせ、cross attention で future dynamic queries を予測する。
- 出力:
  - `future_dynamic_queries [B, H, Q, D]`

### TokenFuser

- `future_dynamic_queries` を current spatial tokens に feature-space で fuse する。
- current spatial tokens に対して query 由来の update を加える。
- 出力:
  - `future_spatial [B, H, N, D]`
  - `fuser_masks [B, H, Q, N]`

### PixelDecoder

- `future_spatial` tokens から future image を復元する。
- 出力:
  - `pred_future [B, H, 3, H_img, W_img]`

### ActionFutureScorer

- v4b で使う ranking head。
- `action_tokens` と `future_dynamic_queries` から scalar の `ranking_score [B]` を出す。
- 正しい action sequence の score が negative action sequence より高くなるように学習する。

## 5. v4a / v4b / v4c の違い

### v4a

- Temporal Dynamic Query WM。
- `ActionFutureScorer` なし。
- temporal dynamic query 構造そのものが future prediction / representation に効くかを見る。
- `LAMBDA_RANK=0.0`。

### v4b

- v4a + `ActionFutureScorer`。
- `ranking_score` を出す。
- `score_pos > score_neg` になるように ranking loss を学習する。
- RFT に使える action-sensitive score を作れるかを見る中心条件。

### v4c

- v4b を RFT に接続する段階。
- `WORLD_REWARD_TYPE=visual / rank_score / hybrid` を比較する想定。
- ただし現時点では、まず v4 core sweep で WM 単体評価を行う段階。

## 6. loss 設計

v4 の total loss:

```text
L_total =
  λ_image     * L_image
+ λ_dynamic   * L_dynamic_roi
+ λ_static    * L_static
+ λ_query     * L_query
+ λ_rank      * L_rank
+ λ_entropy   * L_entropy
+ λ_diversity * L_diversity
```

各 loss の意味:

- `L_image`: `pred_future` と GT future の画像 MSE。
- `L_dynamic_roi`: dynamic mask 領域の MSE。動いている領域を重視する。
- `L_static`: 静的領域を壊さないための loss。
- `L_query`: predicted `future_dynamic_queries` と GT-derived future queries の MSE。
- `L_rank`: `score_pos > score_neg` にする ranking loss。
- `L_entropy`: mask を局所化するための entropy loss。
- `L_diversity`: query 同士が同じ場所を見ないようにする diversity loss。

注意:

- v4 の主目的は Full MSE 最小化ではない。
- `L_rank`, `L_query`, mask entropy/diversity が、RFT に効く representation を作れているかが重要。

## 7. mixed negative の修正内容

v4b の ranking 学習では negative action が重要。

当初、mixed negative の key が以下のようになっていた。

```text
same_task:0.5,temporal_permutation:0.25,zero_random:0.25
```

しかし `V4Collator` が期待する key は `same_task_other_window` だった。

そのため、`same_task_other_window` が 0% として扱われ、`temporal_permutation` / `zero_random` に偏る可能性があった。pilot の `neg_task_match_rate mean=0.25` も、この疑いと整合的だった。

修正後:

```text
NEGATIVE_MIX=same_task_other_window:0.5,temporal_permutation:0.25,zero_random:0.25
```

現在の方針:

- config は canonical key の `same_task_other_window` に統一。
- `V4Collator` では `task_id` を使って同一 task 内の negative action を選ぶ。
- `task_id` がない場合は、誤って同一 task 扱いしないように unique negative sentinel を使う。
- `same_task` が混入した場合は alias として `same_task_other_window` に正規化し、warning を出す。
- negative 診断として以下をログする。
  - `neg_task_match_rate`
  - `negative_type_counts`
  - `num_batch_roll_fallback` / `batch_roll_fallback`

## 9. 各 sweep 軸の意味

### motion bias の効果

比較:

- `v4a_q8_k2_nomotion`
- `v4a_q8_k2_motion`

分かること:

- `||z_t - z_{t-1}||` を使う motion bias が dynamic query の抽出を改善するか。
- 動いている領域に mask が寄り、`dynamic_mask_entropy` / `dynamic_mask_overlap` / LPIPS が改善するか。

### dynamic query 数 Q の効果

比較:

- `v4a_q8_k2_motion` vs `v4a_q16_k2_motion`
- `v4b_q8_k2_rank1_motion` vs `v4b_q16_k2_rank1_motion`

分かること:

- Q=16 に増やすことで表現力が上がるか。
- 逆に query が冗長になり、`dynamic_mask_overlap` / `fuser_mask_overlap` が悪化しないか。
- scorer ありでも Q 増加が `pairwise_acc_score` / `score_gap_mean` を改善するか。

### history length K の効果

比較:

- `v4a_q8_k2_motion` vs `v4a_q8_k3_motion`
- `v4b_q8_k2_rank1_motion` vs `v4b_q8_k3_rank1_motion`

分かること:

- K=3 の長い history が temporal residual query に効くか。
- context gap の複雑さが増えて逆に悪化しないか。
- scorer ありで K=3 が ranking signal を改善するか。

### ActionFutureScorer / ranking loss の効果

比較:

- `v4a_q8_k2_motion`
- `v4b_q8_k2_rank1_motion`

分かること:

- ActionFutureScorer を追加することで action-sensitive ranking signal が立つか。
- ranking loss が WM 本体を壊さず、`pairwise_acc_score` / `score_gap_mean` を改善するか。

### lambda_rank の効果

比較:

- `v4b_q8_k2_rank05_motion`
- `v4b_q8_k2_rank1_motion`
- `v4b_q8_k2_rank2_motion`

分かること:

- ranking loss が弱すぎる / ちょうどよい / 強すぎる のどこにあるか。
- rank2 で score は伸びるが copy-current collapse や mask collapse が出るなら強すぎる。
- rank05 で score が立たないなら弱すぎる。

### negative type の効果

比較:

- `v4b_q8_k2_rank1_motion`
- `v4b_q8_k2_rank1_mixedneg`

分かること:

- same-task only negative と mixed negative のどちらが RFT 接続前に頑健か。
- mixed negative で `score_gap_mean` と `pairwise_acc_score` が保たれるなら、RFT reward として使いやすい可能性が高い。

## 12. 評価で見るべき指標

### 主指標

- `pairwise_acc_score`
  - scorer が正しい action を negative action より高く評価できているか。
  - 0.5 がランダム目安。0.6 以上を一つの目標にする。
- `score_gap_mean`
  - `score_correct - score_negative` の平均。
  - 0 より大きいことが重要。
- `reverse_windows_score`
  - score が逆転した window 数。
  - 少ないほどよい。
- `pairwise_acc_lpips`
  - GT action の reconstruction が shuffled action より LPIPS 的によいか。
- `lpips_gap_mean`
  - `shuffled_lpips - correct_lpips`。
  - 大きいほど action による future prediction の差が出ている。
- `action_correct_best_rate`
  - action 条件の中で correct action が最良になった割合。あれば RFT候補選択に有用。

### negative 診断

- `neg_task_match_rate`
  - negative が positive と同じ task から取れている割合。
  - 低すぎると cross-task easy negative に偏る。
- `negative_type_counts`
  - same_task / temporal_permutation / zero_random の内訳。
  - mixed negative が意図通りになっているか確認する。
- `batch_roll_fallback`
  - same-task candidate が batch 内にない時の fallback。
  - 増えすぎる場合は batch size / task_id / sampler を疑う。

### mask 診断

- `dynamic_mask_entropy`
  - dynamic query mask の集中度。低いほど局所化。
- `dynamic_mask_overlap`
  - query 同士が同じ場所を見ていないか。低いほど多様。
- `dynamic_mask_max`
  - mask の peak 強度。
- `fuser_mask_entropy`
  - TokenFuser の attention 集中度。
- `fuser_mask_overlap`
  - TokenFuser 側の query overlap。

### copy-current 診断

- `full_mse_over_copy_current_mse`
  - 1.0 以上だと copy-current より悪い可能性がある。
- `residual_abs_mean`
  - residual がほぼ 0 に潰れていないか。
- `residual_abs_max`
  - residual の最大値。暴れていないか。

### loss

- `loss_image`
- `loss_dynamic`
- `loss_static`
- `loss_query`
- `loss_rank`
- `loss_entropy`
- `loss_diversity`

見るポイント:

- `loss_rank` が下がっているか。
- `loss_query` が崩れていないか。
- `loss_entropy` / `loss_diversity` が mask/query collapse の兆候を出していないか。
- ranking score だけ伸びて `full_mse_over_copy_current_mse` が悪化していないか。

## 13. RFT候補選択方針

v4 core sweep 後、RFT に進める候補は最大 2〜3 条件に絞る。

候補選択基準:

- `pairwise_acc_score` が高い。
- `score_gap_mean > 0`。
- `reverse_windows_score` が少ない。
- mixed negative で破綻しない。
- copy-current collapse していない。
- mask/query collapse していない。
- v3 最良条件より action ranking signal が改善している。

候補になりやすい条件:

- `v4b_q8_k2_rank1_motion`
  - v4b 標準。第一候補。
- `v4b_q8_k2_rank1_mixedneg`
  - mixed negative で頑健なら第二候補。
- `v4b_q16_k2_rank1_motion`
  - Q=16 が明確に良ければ追加。
- `v4b_q8_k3_rank1_motion`
  - K=3 が明確に良ければ追加。

候補選択の目安:

```text
pairwise_acc_score > 0.55
score_gap_mean > 0
full_mse_over_copy_current_mse < 1.0 付近
dynamic_mask_overlap / fuser_mask_overlap が高すぎない
neg_task_match_rate が十分
```
